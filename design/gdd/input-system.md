## A. 概覽 (Overview)

殲獸戰機是**手機優先（Mobile-primary）、PC 次之**的多平台彈幕射擊遊戲。輸入系統必須同時滿足三種輸入方案——觸控（Touch）、鍵盤＋滑鼠（KB+Mouse）、手柄（Gamepad）——且三種方案在任何難度階的輸出能力完全等價，徹底落實「**難度是門，不是牆**」的設計支柱。

主武器（Primary Weapon）**自動開火**，不需玩家主動輸入。玩家主動管理的動作有且只有四個核心輸入：**移動（Move）、副武器開火（FireSecondary）、L3 蓄力（ChargePrimary，僅裝備 L3 波動砲時）、暫停與 UI 操作（Pause/UI）**。

武器切換**不是輸入動作**：武器由場地掉落的武器莢艙（Weapon Pod）接觸後自動換裝，與玩家輸入無關。本文件不定義武器切換按鍵。

> **[最高風險] 觸控彈幕手感尚未驗證（Touch Bullet-Hell Feel — UNVALIDATED）**
>
> 現有原型（`prototypes/weapon-feel-concept/prototype.html`）的觸控輸入採「游標直接映射（direct cursor-to-ship mapping）」——手指位置即為船艦目標位置，無偏移補償。Sky Force Reloaded / Anniversary 以「相對手指偏移拖曳（relative finger drag offset）」解決「手指遮蔽船艦/判定點」的核心問題，但此機制在本作的彈幕密度與部位破壞互動下是否感覺良好，**尚未有任何實機觸控原型驗證**。在進入 MVP 觸控實作前，必須建立專用的觸控輸入原型並完成 playtest（見 L.1）。

## B. 玩家幻想 (Player Fantasy)

**目標體驗**：玩家感受的是「精密手術刀」的控制精度——小機體彷彿是手指的延伸，在密密麻麻的敵彈縫隙中滑動，每次位移都精確無誤。核心幻想「**以小搏大、以智取勝**」要求輸入系統達到四個品質目標：

- **可預測性（Predictability）**：船艦每次都到達玩家「意圖的位置」，而非手指的實際物理落點。
- **可讀性（Legibility）**：船艦本體與其判定點（hitbox dot）永遠對玩家可見，不被手指、按鈕或 HUD 遮蔽。
- **即時確認（Immediacy）**：副武器開火、L3 蓄力完成、L3 釋放等每一個輸入動作，都有明確的視覺＋音效回饋，讓玩家立即確認「我的輸入被系統接受了」。
- **跨方案等價（Cross-Scheme Parity）**：三種輸入方案的玩家都能在最高難度下破壞所有巨獸部位，沒有任何方案因輸入機制形成事實上的難度壁壘（見 I 章）。

## C. 輸入動作對照表 (Input Action Map)

下表定義所有**抽象動作（Abstract Actions）**及三種方案的對應輸入。

| 抽象動作 | 描述 | 觸控 (Touch) | 鍵盤＋滑鼠 (KB+Mouse) | 手柄 (Gamepad) |
|---|---|---|---|---|
| **移動 Move** | 控制船艦二維位置（類比，全方向） | 全螢幕相對拖曳＋垂直偏移（見 D.1） | 滑鼠游標跟隨（主要）；WASD / 方向鍵（替代） | 左搖桿（Left Stick，類比） |
| **副武器開火 FireSecondary** | 發射當前副武器（消耗一發彈匣額度，尊重換彈狀態） | 右側固定按鈕（單次點擊） | 左鍵點擊 / 空白鍵 | 右扳機（RT / R2）或 A 鈕 |
| **L3 蓄力 ChargePrimary** | 僅裝備 L3 波動砲時可用；長按 ≥1.5 秒後釋放觸發全幅震波；短觸由自動開火處理（弱脈衝，無需玩家操作） | 右側蓄力按鈕長按（僅 L3 裝備時顯示） | 右鍵長按 / Z 鍵長按 | 左扳機（LT / L2）長按 |
| **L3 釋放 ReleasePrimary** | 達到蓄力閾值後放開，觸發全幅震波 | 放開蓄力按鈕 | 放開右鍵 / 放開 Z 鍵 | 放開 LT |
| **暫停 Pause** | 暫停遊戲，顯示暫停選單 | 右上角暫停按鈕 | Esc / P 鍵 | Start / Menu 鈕 |
| **UI 確認 UIConfirm** | 選單/Loadout 選擇確認 | 點擊選項 | Enter / 左鍵 | A 鈕 |
| **UI 取消 UICancel** | 返回上層選單 | 點擊返回 | Esc / Backspace | B 鈕 |
| **UI 導航 UINavigate** | 在選單選項間移動游標 | 點擊目標 | 方向鍵 / Tab | 左搖桿 / D-Pad |

- 武器切換（Weapon Switch）：無對應按鍵。拾取武器莢艙即換裝，玩家可迴避莢艙以保留當前武器。
- 主武器開火（Primary Fire）：主武器自動開火，無需輸入。

## D. 觸控方案 (Touch Scheme) — 主要平台，待驗證

> **[UNVALIDATED — 需要觸控原型]** 本章所有設計均為**假說（Hypothesis）**，需透過實機觸控 playtest 驗證。現有 HTML 原型的觸控邏輯採直接游標映射，不反映本章規格。在驗證前，本章設計不進入正式實作。

### D.1 移動：相對手指偏移拖曳（Relative Finger Offset Drag / Sky Force 方案）

**問題：手指遮蔽問題（Occlusion Problem）**

觸控彈幕遊戲的核心 UX 挑戰：若船艦直接跟隨手指絕對位置，玩家手指會持續遮蔽：
1. 船艦本體（喪失位置感）
2. 船艦判定點（hitbox dot）——在彈幕閃避中是致命的
3. 即將到來的前方彈幕

**解法：相對偏移拖曳**

移動模型如下三個步驟：

**步驟 1 — 觸控開始（TouchStart）**：記錄手指落下位置為「錨點（Anchor Point）」。船艦**不跳到手指位置**，繼續停在當前位置。當前船艦位置記憶為偏移起點（Offset Origin）。

**步驟 2 — 拖曳中（TouchMove）**：船艦的目標位置跟隨手指的**位移向量（delta vector）**：

```
目標位置(t) = 偏移起點 + (當前手指位置 - 錨點位置)
             + vec2(0, −touch_visual_offset_y)   ← 固定向上視覺偏移
```

船艦不跟隨手指的絕對座標，而是跟隨相對移動量。`touch_visual_offset_y` 確保船艦始終顯示在手指正上方，手指不遮蔽船艦。

**步驟 3 — 跟隨平滑（Follow Smoothing）**：船艦以指數平滑（Exponential Lerp）接近目標位置，而非瞬間跳到目標：

```
船艦位置(t) = lerp(船艦位置(t-1), 目標位置(t), touch_follow_lerp × dt_normalized)
```

平滑使急速抖動不直接傳入船艦，同時保留精確感。此係數的最佳值是 playtest 的主要驗證對象。

**其他規格**：
- **拖曳可從螢幕任意位置開始（Drag Anywhere）**：整個螢幕（除右側按鈕覆蓋區域）皆為有效拖曳起點，不需在特定位置起始——降低新手學習門檻。
- **觸控死區（Dead Zone）**：手指從錨點移動 ≤ `touch_dead_zone_px` 時，船艦不動，防止靜止觸摸產生微抖動。
- **場地邊界夾取（Boundary Clamp）**：船艦位置 clamp 於場地邊界內（縮 `ship_margin_px`），手指可移動超出邊界但船艦不越界。
- **觸控釋放（TouchEnd）**：手指離開螢幕後，船艦停在當前位置（不歸位），保持靜止直到下次觸控。

### D.2 副武器開火：專用固定按鈕（Dedicated On-Screen Button）

**方案選擇：固定右側按鈕（推薦），而非「第二拇指任意觸控區（Two-Thumb Tap Zone）」**

| 評估面向 | 固定右側按鈕（推薦） | 第二拇指任意觸控區 |
|---|---|---|
| **視覺可辨性（Affordance）** | 明確；玩家一眼知道在哪裡 | 無視覺錨點，高認知負擔 |
| **高壓誤觸率** | 低；按鈕邊界固定明確 | 高；彈幕高壓時易意外觸碰移動區 |
| **Fitts's Law** | 固定大目標，平均命中時間短 | 目標位置不固定，反應時間不穩定 |
| **單手遊玩** | 支援（見 H.3） | 不支援，強制雙手 |
| **開火節奏可靠性** | 高；每次觸碰有確定回饋 | 低；不確定是否成功觸發 |

**按鈕規格**：
- 位置：螢幕右下角，距邊緣 `ui_button_margin_pt`
- 尺寸：最小 `ui_button_min_size_pt` = 60×60 pt（符合 Apple/Google HIG 觸控目標最低要求）
- 按下回饋：視覺縮放＋音效確認（由 VFX/SFX 系統負責渲染）

### D.3 L3 蓄力：情境式蓄力按鈕（Context-Sensitive Charge Button）

L3 蓄力動作**僅在玩家裝備 L3 波動砲時需要存在**。採**漸進揭露（Progressive Disclosure）**設計，避免常態顯示不相關的 UI 元素：

- **非 L3 裝備時**：蓄力按鈕區域隱藏，螢幕保持乾淨，右側只有副武器按鈕。
- **裝備 L3 時**：螢幕右上方出現「蓄力 CHARGE」按鈕，200 ms 淡入動畫。
- **操作邏輯**：長按蓄力按鈕 ≥ `l3_charge_time`（1.5 秒）→ 顯示「READY」→ 放開按鈕 → 觸發全幅震波並進入 2 秒冷卻。蓄力不足即放開 → 僅輸出弱脈衝（不另需輸入；與自動開火的弱脈衝相同效果）。
- **蓄力視覺回饋**：船艦周圍蓄力環成長動畫（已在 HTML 原型驗證）；達到蓄力閾值後環轉為白色並顯示「READY」文字標示（**文字，不依賴顏色**）。

**物理衝突分析**：蓄力按鈕位於右上角，副武器按鈕位於右下角，移動拖曳在左半及全螢幕區。空間分離設計使三個操作在物理上不重疊。但需注意：蓄力期間右拇指被「鎖定」在蓄力按鈕，**無法同時開火副武器**——這是刻意的戰術取捨（蓄力 OR 飛彈），但需 playtest 確認是否造成挫敗感。

> **[假說，需驗證]** 蓄力中斷寬限（Interrupt Grace Period）：若玩家在蓄力進行中因閃避需要短暫放開蓄力按鈕，放開後 `l3_charge_interrupt_grace_period`（0.3 秒）內重新按下應繼續計時，不重置進度。此設計在 playtest 前是未驗證假說。

### D.4 手指遮蔽問題對策一覽

| 問題 | 對策 | 調校旋鈕 |
|---|---|---|
| 手指遮住船艦本體 | 相對偏移：船艦在手指正上方固定距離 | `touch_visual_offset_y` |
| 手指遮住判定點（hitbox dot） | 偏移確保判定點位於手指上方 | `touch_visual_offset_y` |
| 手指遮住前方即將到來的彈幕 | 偏移使船艦領先於手指，視野較清晰 | `touch_visual_offset_y` |
| 靜止觸摸產生微抖動 | 拖曳死區 | `touch_dead_zone_px` |
| 急速拖曳傳入不穩定位移 | 指數平滑跟隨 | `touch_follow_lerp` |
| 船艦移出場地邊界 | 場地邊界夾取 | `ship_margin_px` |

## E. 鍵盤＋滑鼠方案 (KB+Mouse Scheme)

### E.1 移動：滑鼠跟隨（Mouse-Follow，主要推薦）

**推薦滑鼠跟隨而非 WASD 為主要方案的理由**：

彈幕射擊的核心技能是「精準微閃避（precise micro-dodging）」。WASD 提供數位八方向輸入；滑鼠提供類比精度二維輸入。在密集彈幕下，滑鼠閃避精確度遠高於鍵盤。參考：Sky Force PC 版採滑鼠跟隨；Drainus 同樣優先滑鼠控制。此方案也與觸控的「類比精度控制」設計理念一致，使跨平台手感設計共享同一基礎模型。

- **游標跟隨**：船艦以平滑 lerp（`mouse_follow_lerp`）移動至滑鼠游標位置。PC 版不需遮蔽補償，無垂直偏移。
- **游標夾界**：船艦 clamp 於場地邊界內；游標本身不被限制，可移動到場地外（不中斷操作）。
- **游標隱藏**：遊戲中隱藏系統游標，以船艦本身作為游標的視覺替代。

**WASD / 方向鍵替代方案（鍵盤唯一方案）**：
- 船艦以 `keyboard_move_speed` 速度持續往按壓方向移動。
- 對角方向同時按壓時，速度向量正規化（Normalize），不超過最大速度。
- WASD 方案必須能在全難度下完成完整遊戲（跨方案等價要求）。

### E.2 副武器開火

- **左鍵點擊（Left Click）**：每次點擊發射一次。連點開火；不支援「長按連發」以避免意外耗彈。
- **空白鍵（Space）**：等效替代鍵，供無滑鼠環境使用。
- 兩者皆可重新映射（見 H.1）。

### E.3 L3 蓄力

- **Z 鍵長按**（主要方案；與 HTML 原型一致，已初步驗證可用）：長按 ≥1.5 秒觸發蓄力，放開釋放震波。
- **右鍵長按（Right Mouse Button Hold）**（替代方案）：允許玩家在不離開滑鼠的情況下蓄力，不需使用鍵盤。
- 兩方案皆可重新映射。蓄力進度視覺回饋（蓄力環 + READY 文字）與觸控版一致。

| **L3 蓄力 ChargePrimary** | 左扳機（LT / L2）長按 | 僅 L3 裝備時生效；其他主武器裝備時此鍵空置 |
| **暫停 Pause** | Start / Menu 鈕 | |

| **UI 取消** | B 鈕 | |
| **UI 導航** | 左搖桿 / D-Pad | |

### F.2 搖桿移動設計

- **速度模式（Velocity Mode）**：搖桿傾斜幅度決定移動速度，而非目標位置。完全傾斜 = `gamepad_max_speed`（280 px/s）。
- **輸入曲線（Input Curve）**：微量搖桿輸入更細膩，避免小角度傾斜產生過快位移：

  ```
  speed = gamepad_max_speed × pow(stick_magnitude, gamepad_input_curve)
  ```

  `gamepad_input_curve` 預設 1.5——輕推搖桿精度提升，完全傾斜達到最大速度。

- **震動回饋（Haptic Feedback）**：部位破壞時觸發短暫震動脈衝（`haptic_break_duration`、`haptic_break_intensity`）。此功能必須可在設定中完全關閉（無障礙要求）。

## G. 手感調校 (Feel Tuning)

### G.1 移動跟隨響應

這是觸控彈幕手感最核心的問題：**船艦跟隨手指的速度，必須在「足夠快以閃避彈幕」和「不過快以至於手抖造成死亡」之間取得平衡。** 此平衡沒有公式解——它必須透過 playtest 找到。

| 參數 | 原型基準值 | 設計意圖 | 驗證狀態 |
|---|---|---|---|
| `touch_follow_lerp` | ~0.92/幀（原型估算） | 平滑但不遲鈍；急速拖曳約 0.1 幀感知延遲 | **未驗證 — 觸控 playtest 必需** |
| `mouse_follow_lerp` | ~0.98/幀（原型估算） | 幾乎即時；保留精確感 | 原型初步可用 |
| `gamepad_max_speed` | 280 px/s | 穿越 320px 寬場地約 1.15 秒 | **未驗證 — 手柄 playtest 必需** |

### G.2 彈幕密度與觸控精度的張力

四階難度主要改變敵彈密度，**不改變輸入系統行為**。輸入系統的責任是確保玩家「因技術失敗，而非因輸入失敗」：

- 觸控精度物理極限：人類觸控輸入誤差約 ±2–3 mm，對應手機螢幕約 ±6–9 px（在 160–240 dpi 裝置上）。
- **彈幕設計約束（非輸入系統直接控制，需與關卡設計協作）**：任何難度階的最窄彈幕縫隙，不應要求 ≤10 px 精度才能通過——這超出了觸控輸入的物理能力上限，違反「難度是門」支柱。

### G.3 L3 蓄力手感目標

L3 蓄力是本遊戲輸入複雜度最高的動作：玩家需同時移動閃避＋維持長按。蓄力手感設計目標：

- **進度清晰**：蓄力環的成長給予明確的「還需要多久」感知，不讓玩家盲目等待。
- **不依賴顏色**：蓄力完成狀態以「READY」文字標示（已在 HTML 原型實作），色弱玩家不受影響。
- **容錯設計**：蓄力中斷寬限（`l3_charge_interrupt_grace_period`）允許短暫放開後繼續計時，減少「因閃避意外打斷蓄力」的挫敗感（此設計為假說，需 playtest 驗證）。

## H. 重新映射與無障礙 (Remapping & Accessibility)

### H.1 全面可重新映射（Full Remapping）

所有動作的按鍵綁定必須可在設定介面中重新映射：

| 方案 | 可映射動作 |
|---|---|
| 鍵盤 | FireSecondary、ChargePrimary / ReleasePrimary、Pause、所有 UI 動作 |
| 滑鼠按鍵 | FireSecondary 與 ChargePrimary 可在左鍵/右鍵/中鍵間互換 |
| 手柄 | 全部動作；左搖桿 X/Y 軸可分別翻轉（Axis Invert） |
| 觸控 | 副武器按鈕位置（左手模式/右手模式切換）；蓄力按鈕位置（左上/右上）；`touch_visual_offset_y` 玩家可調 |

**映射衝突檢測**：重新映射介面偵測兩個不同動作被映射到相同按鍵的情況，並以警告提示玩家解決；不允許儲存衝突映射。

### H.2 L3 蓄力：Hold 模式 vs. Toggle 模式

長按（Hold-to-charge）可能對部分玩家造成手指疲勞，或因關節炎、運動障礙而無法持續按壓。提供兩種蓄力操作模式：

- **Hold 模式（預設）**：長按觸發蓄力，持續按住計時，放開即釋放震波。符合直覺的標準操作。
- **Toggle 模式（無障礙替代）**：第一次按下開始蓄力計時，計時結束自動釋放震波，**不需持續按壓**。適合關節炎、手部疲勞、或需要單手/有限手部活動度遊玩的玩家。

兩種模式功能等價，不影響遊戲性。

### H.3 單手遊玩考量（One-Handed Play）

觸控方案的按鈕布局需考慮單手拇指操作的可能性：
- 副武器按鈕預設在右下角；設定中提供「左手模式」，將副武器按鈕移至左側。
- 按鈕尺寸（≥60×60 pt）確保拇指在拖曳移動時能快速切換至按鈕，不需精確定位。
- 單手拇指同時控制移動和偶爾點擊副武器的可行性需在 playtest 中以單手方式測試。

### H.4 不依賴精確多點觸控（No Precise Multi-Touch Required）

核心遊玩的任何情境都不要求同時精確定位兩個觸控點：
- 移動拖曳（左/全螢幕）與副武器按鈕（固定大目標）的同時操作允許且為常態，但副武器按鈕大到不需精確定位右拇指。
- 最複雜情境（移動 + 副武器 + L3 蓄力）中，三個觸控目標皆為大固定目標，不要求多點精確定位。此情境須在 playtest 中特別觀察。

### H.5 無障礙通用檢核（Accessibility Checklist）

- [x] 僅使用鍵盤可完成全遊戲（WASD + 空白鍵 + Z 鍵）
- [x] 僅使用手柄可完成全遊戲
- [ ] 所有 UI 文字在最小字型下可讀（由 UI 系統負責；輸入系統需確保觸控按鈕標籤依螢幕密度正確縮放）
- [x] 不依賴顏色辨識操作狀態（L3 蓄力完成以「READY」文字標示；副武器換彈以文字＋進度條雙軌提示）
- [ ] 無閃爍內容超過 3 Hz 閾值（蓄力完成閃光效果需視覺審核確認；由 art-director 協作評估）
- [ ] 教學提示文字提供無障礙字幕替代（無對話；教學文字即字幕，由 UI 系統負責）
- [ ] UI 在所有支援解析度正確縮放（由 UI 系統負責；輸入系統確保觸控目標面積依 pt 而非 px 定義）
- [x] 震動回饋可在設定中完全關閉

## I. 跨方案等價 (Cross-Scheme Parity)

**核心承諾**：三種輸入方案對全部遊戲內容的存取權完全相同——沒有任何部位、難度、或武器組合只有特定輸入方案能達成。具體要求：

1. **破壞部位能力等價**：觸控、KB+Mouse、手柄玩家均可在任何難度下破壞所有巨獸部位。若某部位需要極端精度（例如小型快速移動弱點），設計需確認三種方案的精度上限都能滿足，不得僅以「PC 玩家容易，手機玩家太難」告終。
2. **L3 蓄力等價**：三種方案在彈幕壓力下完成「蓄力→釋放」的操作成功率，任一方案不得顯著低於其他方案（目標：各方案 ≥80%，見 L.2）。若某方案成功率過低，調整**該方案的輸入設計**，而非放寬 L3 的蓄力時間（避免影響其他方案的平衡）。
3. **副武器節奏等價**：彈匣耗盡→換彈的時機窗口在三種方案下感覺一致。
4. **難度是門的落實**：四階難度的全部內容對三種輸入方案均完全開放，不得因輸入機制形成事實上的隱性難度差異。

## K. 調校旋鈕 (Tuning Knobs)

**所有數值必須存放於外部設定檔（`assets/data/input/input_settings.json`），禁止在程式碼中硬編碼。**

### K.1 觸控調校旋鈕

| 旋鈕名稱 | 預設值 | 安全範圍 | 類型 | 說明 |
|---|---|---|---|---|
| `touch_visual_offset_y` | 80 px（遊戲座標） | 40–120 | 手感 | 船艦在手指正上方的垂直偏移距離；值越大手指遮蔽越少，但移動感越「飄」；需 playtest 校準 |
| `touch_dead_zone_px` | 4 px（螢幕物理像素） | 2–10 | 手感 | 拖曳死區半徑；低值靈敏但易抖動，高值穩定但起步遲鈍 |
| `touch_follow_lerp` | 0.92（每幀比例，60fps 基準） | 0.80–0.99 | 手感 | 移動平滑指數係數；低值偏遲鈍，高值偏即時；**未驗證，playtest 主要目標** |
| `touch_button_min_size_pt` | 60 pt | 48–80 | 閘門 | 觸控按鈕最小尺寸（pt，不因螢幕密度改變物理面積）；不得低於 48 pt（Apple HIG 最低要求） |
| `touch_button_margin_pt` | 16 pt | 8–24 | 手感 | 按鈕距螢幕邊緣距離 |
| `l3_charge_time` | 1.5 秒 | 1.2–2.0 | 閘門 | L3 蓄力所需長按時間（與 weapon-system.md `l3_charge_time` 旋鈕同步，必須保持一致） |
| `l3_charge_interrupt_grace_period` | 0.3 秒 | 0.0–0.5 | 手感 | 蓄力中斷後不重置計時的寬限時間；**假說——需 playtest 驗證是否必要** |
| `ship_margin_px` | 12 px（遊戲座標） | 8–20 | 閘門 | 場地邊界內縮量（船艦中心點 clamp 邊界） |

### K.2 鍵盤＋滑鼠調校旋鈕

| 旋鈕名稱 | 預設值 | 安全範圍 | 類型 | 說明 |
|---|---|---|---|---|
| `mouse_follow_lerp` | 0.98（每幀比例，60fps 基準） | 0.90–1.00 | 手感 | 滑鼠跟隨平滑係數；1.0 = 即時跟隨 |

## L. 驗收標準 (Acceptance Criteria)

### L.1 [阻斷] 觸控手感驗證（Touch Feel Validation — BLOCKING pre-MVP）

> 此為**本文件最高優先級的未解風險**。以下四項標準必須全部通過，觸控實作才能進入 MVP。

- [ ] **觸控專用原型建立**：在 `prototypes/weapon-feel-concept/prototype.html` 基礎上，實作 D.1 的相對偏移拖曳模型（`touch_visual_offset_y`、`touch_dead_zone_px`、`touch_follow_lerp`），可在實機（iOS/Android 手機）觸控測試。
- [ ] **遮蔽問題解決**：5 位測試者（含至少 1 位首次遊玩觸控 shmup 的新手）在全程測試中反應「我始終知道船艦在哪裡」，且能說出判定點的位置。5/5 通過為目標，4/5 為最低接受門檻。
- [ ] **彈幕閃避可行性**：5 位測試者在最低難度下，觸控操控完成第一個 Boss 戰（至少一次），平均死亡次數 ≤5。若平均死亡次數 >10，需調整 `touch_visual_offset_y` 與 `touch_follow_lerp` 後重新測試。
- [ ] **L3 蓄力觸控可行性**：5 位測試者中至少 4 位能在 Boss 戰壓力下（有彈幕回避需求）成功觸發 L3 全幅震波至少一次。

### L.2 [阻斷] 跨方案等價（Cross-Scheme Parity — BLOCKING pre-Vertical Slice）

- [ ] 三種輸入方案的玩家各自完成原型 Boss 戰（標準難度），觸控方案的部位破壞數中位數 ≥ KB+Mouse 方案中位數的 0.8×（觸控不得顯著劣於滑鼠）。
- [ ] 三種方案的玩家在計時測試中成功觸發 L3 全幅震波的成功率各自 ≥ 80%（10 次嘗試中 ≥8 次成功）。

### L.3 [阻斷] 鍵盤單一方案完整遊玩（Keyboard-Only — BLOCKING）

- [ ] WASD + 空白鍵 + Z 鍵（不使用滑鼠）可完成任一難度的完整遊戲流程，從 Loadout 選擇到擊倒 Boss。

### L.4 [阻斷] 手柄完整遊玩（Gamepad-Only — BLOCKING）

- [ ] 手柄（左搖桿 + RT + LT + Start）可完成任一難度的完整遊戲流程。

### L.5 [阻斷] 無障礙基線（Accessibility Baseline — BLOCKING）

- [ ] L3 蓄力的 Toggle 模式可用，且與 Hold 模式達到相同的震波發射效果。
- [ ] 震動回饋可在設定中完全關閉，關閉後不影響任何遊戲性（視覺回饋完整替代觸覺回饋）。
- [ ] 蓄力完成狀態的「READY」文字標示（或等效非顏色辨識手段）在全部實作中保留，不得以「顏色變化」作為唯一提示。

### L.6 [建議] L3 蓄力中斷寬限（Advisory）

- [ ] 玩家在蓄力進行到 1.0 秒時短暫放開蓄力輸入，在 0.3 秒寬限期內重新長按，蓄力繼續從 1.0 秒計時（不歸零）。驗證方式：10 次手動計時測試，記錄成功率。若 ≥8/10 成功，此功能確認有效；若成功率過低，移除此設計。

### L.7 [建議] 觸控按鈕誤觸率（Advisory）

- [ ] 副武器按鈕在彈幕高壓情境下（測試者主動執行連續閃避動作）的誤觸率 ≤10%（每 10 次開火操作中，≤1 次意外觸發非預期動作）。
